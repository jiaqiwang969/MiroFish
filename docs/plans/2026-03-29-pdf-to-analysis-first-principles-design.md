# PDF 到分析结论的第一性原理讲义设计

**目标**

面向“有一定技术基础的产品/研究人员”，产出一份可直接教学使用的中文系统讲义，解释本项目如何从一份 PDF 文档出发，经过文本抽取、结构化、本体设计、图谱构建、Agent 化、仿真和报告生成，最终得到一份舆情趋势分析结论。

本讲义必须满足以下约束：
- 使用我们本次武汉大学案例的真实配置与真实运行数据
- 以“第一性原理”而不是“功能说明”来组织内容
- 作为正式教学材料，具有清晰的章节结构、插图、表格和实验附录
- 以 `XeLaTeX` 输出中文 PDF

**受众**

- 有一定技术基础，但不是纯工程实现者
- 希望理解“为什么系统这样设计”
- 需要把案例复用到产品设计、研究分析、方案评审或教学培训中

**核心设计原则**

1. 先讲抽象，再讲实现，再回到案例。
2. 先解释“为什么不能直接总结 PDF”，再解释为什么必须构造中间表示。
3. 把“图谱”定义为系统的可检索、可推演、可复用中间世界模型。
4. 把“仿真”定义为在结构化世界模型上运行的多主体观点演化过程。
5. 把“报告”定义为二次推理产物，而不是一次普通摘要。
6. 速度、质量、可解释性三者同时讨论，不能只讲功能正确。

**讲义主线**

讲义采用一条从输入到输出的单主线：

`PDF -> 文本 -> chunk -> ontology -> graph -> entities -> agent profiles -> social simulation -> report`

每一层都回答四个问题：
- 输入是什么
- 输出是什么
- 这一层解决了什么问题
- 这一层的代价、风险和瓶颈是什么

**案例数据边界**

讲义直接写入本次真实案例的关键参数与结果：
- 输入 PDF：`武汉大学品牌声誉深度分析报告.pdf`
- 文本长度：`20862`
- 模型：`gpt-5.4-mini`
- 图谱后端：`graphiti`
- embedding 模型：`BAAI/bge-m3`
- chunk 参数：`6500 / 650`
- build batch size：`2`
- 图谱规模：`4 chunks / 64 nodes / 80 edges`
- 模拟配置：`59 个 Agent`
- 时间数据：建图、准备、仿真、报告各阶段耗时
- 质量信号：报告阶段本地搜索 21 次，其中 18 次为 0 命中

讲义中不写入敏感凭据，不展示 API key。

**文档结构**

1. 引言：为什么“从 PDF 到分析”不是摘要问题
2. 第一性原理：信息压缩、结构化、中间表示、可检索、可仿真
3. 系统总览：全链路架构与四类状态转换
4. 第一步：PDF 如何变成可计算文本
5. 第二步：为什么要分块，以及 chunk 的边界条件
6. 第三步：本体生成，本质上在做什么
7. 第四步：图谱构建，本质上在做什么
8. 第五步：为什么要从实体图谱转成 Agent 世界
9. 第六步：社会仿真如何把结构化记忆转成动态观点演化
10. 第七步：报告生成为什么是“检索 + 采访 + 归纳”的二阶推理
11. 第八步：速度瓶颈与质量瓶颈的统一解释
12. 案例复盘：武汉大学案例的真实数据推导
13. 参数调优方法论
14. 结论：什么叫“可被推演的中间世界模型”
15. 附录：真实日志、配置与结果摘要

**插图策略**

讲义中至少包含以下插图：
- 图 1：从 PDF 到分析结论的全链路流程图
- 图 2：信息在各层的“压缩与重表达”示意图
- 图 3：本次运行的时间分布条形图
- 图 4：chunk 大小、速度、检索命中率之间的权衡图
- 图 5：报告阶段的“图谱检索 + Agent 采访 + 写作”闭环图

所有插图优先使用 `TikZ/PGFPlots` 直接在 LaTeX 中生成，避免额外二进制素材依赖。

**版式策略**

- 使用 `ctexbook` 和 `XeLaTeX`
- 中文主字体优先使用 macOS 常见字体
- 强调教学资料气质，而不是学术论文气质
- 使用摘要框、结论框、定义框、案例框提升可读性
- 允许较长正文，但每章必须有“本章核心结论”

**实现边界**

- 不修改现有业务逻辑
- 仅新增讲义文档与编译产物目录
- 编译命令固定为 `xelatex`
- 最终交付应包含可重新编译的 `.tex` 源文件和已编译完成的 `.pdf`

**成功标准**

- 讲义可以直接被产品/研究人员阅读，不依赖额外口头解释
- 所有关键概念都能在案例中找到对应事实
- PDF 可成功通过 `xelatex` 编译
- 讲义正文长度达到系统版标准，预计 20-30 页
